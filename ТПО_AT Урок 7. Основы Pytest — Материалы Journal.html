Знакомство с Pytest

Pytest — это фреймворк для автоматизированного тестирования в Python. Он используется для написания и запуска тестов и особенно популярен за счёт своей простоты, расширяемости и удобства в использовании. Pytest позволяет писать как простые модульные тесты, так и сложные интеграционные тесты с использованием настройки предусловий, постусловий, окружения и параметризации.

Pytest отличается от многих других тестовых фреймворков тем, что делает акцент на читаемость и лаконичность кода. Вместо громоздких конструкций и строгой структуры, как, например, в unittest, (с которым мы то же познакомимся в рамках этого курса) здесь можно использовать обычные функции и стандартные assert-выражения  булевы выражения в Python, которые проверяют, является ли условие истинным или ложным). Это снижает порог входа для начинающих разработчиков и тестировщиков, а так же делает тесты более понятными и поддерживаемыми на больших проектах.

Одна из ключевых особенностей pytest — мощная система фикстур. Фикстуры позволяют определять повторно используемые элементы окружения, которые автоматически передаются в тестовые функции. Благодаря этому достигается высокая степень повторного использования кода и отделение логики подготовки теста от самих проверок. Система фикстур в pytest устроена гибко и поддерживает вложенность, переопределение и управлять тем, как долго они живут — например, для одного теста, модуля или всей сессии (scope).

Еще одной сильной стороной фреймворка является его экосистема плагинов. Существует множество расширений, позволяющих интегрировать pytest с другими инструментами: CI/CD-системами, фреймворками веб-разработки, средствами сбора покрытия кода и генерации отчётов. Кроме того, сам pytest можно легко расширить, написав собственные решения или плагины под нужды конкретного проекта.

Pytest также предоставляет удобные инструменты для группировки и маркировки тестов. Это позволяет запускать только нужную часть тестов, например, только «медленные» или «критические». Такая гибкость особенно полезна в больших проектах с сотнями или тысячами тестов, где важно уметь быстро изолировать нужные проверки.

Наконец, важно отметить, что pytest — это активно развиваемый open-source проект с большим и вовлечённым сообществом. Он широко используется в индустрии, поддерживается многими крупными компаниями и фреймворками (например, FastAPI, Django и Flask через плагины), а также регулярно обновляется с учётом современных практик и потребностей разработчиков.

Установка и настройка pytest

Установка через pip

Самый распространённый способ установить pytest — через пакетный менеджер pip. Команда выполняется в терминале с помощью команды pip install pytest

Эта команда устанавливает последнюю стабильную версию pytest, доступную в Python Package Index (PyPI). Установка происходит глобально или в рамках активированного виртуального окружения, если оно используется.

Виртуальная среда — это изолированное пространство, где можно устанавливать библиотеки и пакеты, не затрагивая основную (глобальную) систему Python. Она создаётся с помощью инструментов вроде venv (встроенный в Python) или virtualenv.

Виртуальная среда в Python обеспечивает изоляцию зависимостей, позволяя каждому проекту использовать свои версии библиотек, не мешая другим. Например, один проект может работать на Django 3.2, а другой — на Django 4.1, и при этом они не будут конфликтовать. Использование виртуальной среды сохраняет чистоту системы. Глобальное окружение Python остаётся стабильным и не перегружено лишними пакетами, что упрощает его поддержку и снижает вероятность ошибок. Кроме того, виртуальная среда обеспечивает повторяемость и переносимость. В ней можно сохранить список всех зависимостей, это позволяет легко развернуть проект на другой машине, наладить совместную работу в команде и повысить надёжность развёртывания в продакшене. Изолированное окружение также повышает безопасность — одна библиотека не сможет повлиять на другие проекты или на систему в целом, а значит, снижается риск уязвимостей. 

Удобнее всего изучать новую библиотеку создав новый проект в PyCharm, обратите внимание на то, что Тип интерпретатора (Interpreter type) выбран сразу с виртуальной средой (project venv). Задаете название новому проекту и нажимаете Create

 

после этого вводите в терминале PyCharm команду pip install pytest и ждете конца установки фрейморка pytest

Запуск первого теста

Особенности создания файла

Для того что бы запустить первый тест, в проекте необходимо создать файл, который будет начинаться с test_»имя файла»

По умолчанию pytest ищет тесты по определённым шаблонам имён. Файлы должны начинаться с test_ или заканчиваться на _test.py.

Примеры допустимых имён: test_math.py, math_test.py. Файлы с именами вроде math.py pytest не распознаёт как тестовые и игнорирует.

Внутри тестовых файлов pytest обнаруживает функции, которые начинаются с test_. Функции с другими именами, например addition() или check_result(), не будут автоматически выполнены.

Также pytest распознаёт классы, имена которых начинаются с Test и которые не содержат метода __init__. Внутри таких классов функции также должны начинаться с test_.

Если эти правила не соблюдаются, pytest не найдёт тесты. В таком случае запуск тестов приведёт к сообщению вроде collected 0 items, и тесты выполнены не будут.

Существуют способы изменить поведение поиска тестов через конфигурационные файлы (например, pytest.ini), но для упрощения работы и совместимости с другими инструментами рекомендуется придерживаться стандартных правил именования.

Самый простой тест

Внутри файла напишем самый простой тест

def test_sum():
     assert 2 + 2 == 4

# или

def test_str():
     assert "example" in "string example"

def — ключевое слово Python, обозначающее определение функции.

test_sum — имя функции. Начинается с test_, чтобы pytest мог автоматически определить её как тестовую. Название может быть произвольным, но должно начинаться с test_.

assert — оператор проверки. Используется для утверждения, что выражение справа истинно. Если результат ложный, тест считается проваленным.

2 + 2 == 4 — выражение, проверяемое тестом. Если оно истинно, тест проходит. Если ложно, возникает ошибка.

Тест проверяет, что результат сложения 2 и 2 равен 4. 

После того как написан первый тест, его надо запустить. Сделать это можно разными способами:

  1. Запустить все тесты в проекте
  2. Запустить конкретный файл
  3. Запустить выборочный тест
  4. Запустить группу тестов

Что бы запустить все тесты в проекте нужно в терминале прописать команду pytest 

Что бы запустить все тесты в конкретном файле нужно в терминале прописать команду pytest «название вашего файла» 

Что бы запустить один тест, нужно в рабочей области с тестами нажать на зеленый треугольник напротив того теста, который нужно запустить.

Что бы запустить группу тестов, нужно пометить эти тесты маркером.

В pytest для маркировки (пометки) тестов используется специальный декоратор @pytest.mark.<имя_маркера>. Маркеры позволяют логически группировать, фильтровать и управлять запуском тестов по категориям.

@pytest.mark.<имя>   #Маркировка
def test_...():
    ...

Тогда запуск такой группы тестов будет прописываться в терминале как «pytest -m smoke»

а если мы, наоборот, хотим исключить какую-либо группу тестов из прогона то нужно будет прописать в терминале «pytest -m «not smoke»»

Однако при использовании пользовательских маркеров без регистрации pytest выдаёт предупреждение PytestUnknownMarkWarning. 

Чтобы избежать этих сообщений и сделать работу с тестами чище, пользовательские маркеры нужно явно регистрировать в конфигурационном файле pytest.ini.

Файл pytest.ini размещается в корне проекта и содержит раздел [pytest], 

в котором можно указать список допустимых маркеров. Например:

[pytest]
markers =
    smoke: дымовые тесты
    db: тесты, взаимодействующие с базой данных
    api: тесты, проверяющие API-эндпоинты

Регистрация маркеров устраняет предупреждения,
документирует доступные пометки и облегчает поддержку проекта.
И как можно видеть, предупреждение, которое было до заведения
нами маркеров — исчезло.

После запуска теста или тестов, в терминале мы увидим результат прогона. 

В данном тесте 2+2 действительно равно 4, значит результат выполнения этого теста passed

В отчете, который отображен в терминале, мы можем видеть что собран «collected» один тест и он пройден, если тестов для прохождения будет больше , то и в терминале отобразится соразмерное количество и процент (на скриншоте это 100%) поделится на все тесты, как показатель прироста прохождения.

Это более явно видно когда мы запрашиваем развернутый отчет «pytest -v»

Если есть необходимость сформировать отчет о тестировании в отдельный файл, то стоит использовать команду «pytest -v test_1.py | tee report.txt»

После прогона тестов в файловой области проекта сформируется файл с отчетом (название файла может быть любое, для примера выбран report.txt)

«Говорящие ошибки»

Представим, что 2+2 по каким то причинам в нашем тесте не равно 4, а равно 3. Тогда наш тест выдаст ошибку — failed

При запуске тестов с использованием pytest в терминале отображается подробный отчёт о результатах, особенно если какой-либо тест завершился неудачно. Такой отчёт включает ключевую информацию, необходимую для анализа и устранения ошибки.

Пример вывода показывает, что тестовая функция test_sum в файле test_1.py не прошла проверку. Указывается конкретное место ошибки — третья строка файла. Далее отображается результат сравнения: фактическое значение (4) не совпало с ожидаемым (3). Отчёт содержит явное сообщение о несоответствии (4 != 3), а также выделяет строчку кода, где произошёл сбой, с использованием стрелки и префикса E для указания на ошибку. В конце резюмируется, что один тест завершился неудачей, и приводится общее время выполнения.

Такой формат отчёта удобен тем, что предоставляет всю нужную информацию компактно и в структурированном виде. 

Давайте более подробно поговорим про возможность assert

assert — это оператор в Python, предназначенный для проверки логических условий во время выполнения программы. Если условие ложно, выполнение прерывается с исключением AssertionError.

«Я ожидаю, что это условие выполняется. Если нет — это ошибка.»

Он используется для проверки корректности выполнения кода, обнаружения логических ошибок на этапе разработки и в тестах для утверждения ожидаемого поведения.

В Pytest оператор assert применяется как основной инструмент для написания проверок. Фреймворк автоматически анализирует выражение внутри assert и при ошибке выводит подробное сообщение, показывающее, какое значение ожидалось и какое получилось на самом деле.

Под «assert-like» в Pytest понимаются дополнительные возможности, расширяющие стандартное поведение assert.

К ним относятся автоматический разбор выражений и вывод детализированной информации при сбое, поддержка сложных логических условий, возможность проверки исключений с помощью конструкции pytest.raises. В отличие от других фреймворков, в Pytest не требуется использовать специальные assert-методы — достаточно обычного оператора assert.

Параметризация тестов

Параметризация в Pytest — это способ запускать один и тот же тест с разными входными данными. Это особенно важно для тестировщика, потому что позволяет удобно покрыть позитивные и негативные кейсы в одном тесте, без дублирования кода.

Вместо того чтобы писать несколько похожих функций, можно задать параметры один раз, и Pytest автоматически выполнит тест для каждого набора данных. Это делает тесты более читаемыми, компактными и легче поддерживаемыми.

Параметризация помогает:

  • Проверить, что функция работает с разными позитивными данными.

  • Убедиться, что при негативных данных поведение остаётся ожидаемым (например, возникает исключение или возвращается ошибка).

  • Существенно сократить количество строк кода при расширении покрытия.

С точки зрения процесса тестирования, параметризация упрощает создание набора типовых сценариев, включая граничные значения, альтернативные форматы входных данных и ошибки пользователя — всё это можно перечислить в одном месте, не копируя тестовую логику.

import pytest

@pytest.mark.parametrize("a, b, expected", [
    (2, 2, 4),   # позитивный кейс
    (1, 1, 2),   # позитивный кейс
    (0, 0, 0),   # граничный случай
    (-1, 1, 0),  # позитивный кейс
    (2, 2, 5),   # негативный кейс — намеренно неверное ожидание
])
def test_sum(a, b, expected):
    assert a + b == expected

Строка @pytest.mark.parametrize("a, b, expected", [...]) означает, что Pytest будет подставлять разные значения в параметры a, b и expected при каждом запуске теста. Эти параметры затем передаются в функцию теста.

Список в квадратных скобках содержит кортежи с входными данными. Каждый кортеж — это один запуск теста. Например, (2, 2, 4) означает, что в тесте переменные будут равны a = 2, b = 2, expected = 4.

Строка @pytest.mark.parametrize("a, b, expected", [...]) относится только к той функции теста, которая идёт сразу после неё.

Декоратор влияет только на одну функцию, под которой написан. Он не действует на весь файл и не распространяется на другие тесты.

Если необходимо параметризовать несколько тестов, нужно отдельно написать @pytest.mark.parametrize(...) перед каждой функцией, к которой это применимо.

Таким образом, один тест запускается несколько раз с разными данными, и можно проверить, как код работает в разных ситуациях, не копируя одну и ту же функцию много раз. Это сокращает количество строк кода и делает тесты более удобными для чтения и поддержки.

Фикстура в pytest

Фикстура в Pytest — это вспомогательная функция, которая автоматически подготавливает данные или окружение, необходимые для выполнения теста. Её задача — вынести повторяющуюся подготовительную логику из тестов, чтобы не дублировать один и тот же код в каждом тесте.

Фикстуры особенно полезны, когда:

  • перед каждым тестом нужно выполнить какие-то действия (например, открыть соединение, создать объект, подготовить файлы);

  • нужно очистить или сбросить состояние после теста;

  • тесты используют одни и те же входные данные или структуру.

Фикстура создаётся при помощи декоратора @pytest.fixture. Чтобы использовать её в тесте, достаточно указать имя фикстуры как аргумент в функции теста — Pytest сам подставит результат выполнения фикстуры.

Фикстуры могут:

  • возвращать любые данные (например, словарь, объект, соединение);

  • использоваться повторно в нескольких тестах;

  • быть зависимыми от других фикстур;

  • автоматически вызываться один раз на файл, модуль, сессию или каждый раз заново — это регулируется параметром scope.

Важное отличие от параметризации: фикстуры используются для подготовки окружения или данных, а параметризация — для запуска одного и того же теста с разными значениями.

Фикстуры делают тесты более читаемыми, удобными в поддержке и структурированными, особенно при большом количестве тестов.

Пример 1: фикстура, возвращающая данные:

import pytest

# фикстура
@pytest.fixture
def test_data():
    return {"a": 2, "b": 3}

# тест с использованием фикстуры
def test_sum(test_data):
    result = test_data["a"] + test_data["b"]
    assert result == 5

Пояснение:

  • Функция test_data помечена как фикстура с помощью @pytest.fixture.
  • Она возвращает словарь с входными данными.
  • Тест test_sum получает эти данные автоматически через аргумент test_data.
  • Фикстура может использоваться в любом количестве тестов, не нужно писать одни и те же данные каждый раз.

Пример 2: фикстура с очисткой ресурсов и использованием scope

import pytest

# Фикстура, которая работает один раз для всех тестов в файле
@pytest.fixture(scope="module")
def setup_and_teardown():
    print(" Подготовка")
    resource = {"connected": True}
    yield resource
    print(" Очистка")
    resource["connected"] = False

# Тест использует фикстуру
def test_connection_active(setup_and_teardown):
    assert setup_and_teardown["connected"] is True

# Второй тест использует ту же фикстуру
def test_connection_still_active(setup_and_teardown):
    assert setup_and_teardown["connected"] is True

Пояснение:

  • scope="module" означает, что фикстура будет вызвана один раз для всех тестов в этом файле.
  • До yield — код, который выполняется перед тестом (настройка).
  • После yield — код, который выполняется после выполнения всех тестов, использующих эту фикстуру (очистка).
  • yield resource в фикстуре Pytest — это ключевой момент, который разделяет подготовку и завершение работы фикстуры.
  • Строка resource = {"connected": True} в примере фикстуры — это просто символическая имитация ресурса, с которым якобы нужно работать в тестах. Например, это может быть подключение к базе данных, к API, или к удалённому серверу. resource — это переменная, в которую сохраняется некий объект (в примере — словарь). {"connected": True} — это сам объект, в данном случае простой словарь, указывающий, что «ресурс подключён».

Такая фикстура полезна, когда нужно, например, подключиться к базе, создать файл или запустить сервер, а затем всё это корректно завершить.

В первом примере фикстура test_data вызывается отдельно для каждого теста, в котором она используется, потому что у неё по умолчанию scope="function".

То есть фикстура не общая для всех тестов, если явно не задан другой scope.

Чтобы фикстура вызывалась только один раз на файл, нужно использовать scope="module", или scope="session" — для одного запуска всей тестовой сессии.

Знакомство с плагинами

Pytest поддерживает систему плагинов, которая позволяет значительно расширить его функциональность. Эти плагины полезны при построении полноценной инфраструктуры тестирования, автоматизации, работе с веб-приложениями, API, отчётами, производительностью и другими задачами.

Ниже приведены наиболее часто используемые плагины, которые применяются в практике тестирования. 

1. pytest-cov
Плагин для измерения покрытия кода тестами (code coverage). Он интегрируется с библиотекой coverage.py и позволяет понять, какие части кода действительно были выполнены во время запуска тестов, а какие — нет. Это помогает улучшить полноту тестов.
Особенности:

  • Показывает процент покрытия по файлам.

  • Можно генерировать отчёты в текстовом, HTML или XML формате.

  • Полезен при построении CI/CD процессов.

2. pytest-xdist
Плагин для параллельного и распределённого запуска тестов. Позволяет выполнять тесты одновременно в нескольких процессах или на нескольких машинах.
Особенности:

  • Ускоряет прогон большого набора тестов.

  • Поддерживает стратегию «каждый тест в отдельном процессе» (опция --boxed).

  • Можно распределять выполнение по разным хостам.

3. pytest-html
Плагин для генерации HTML-отчётов по результатам тестов.
Особенности:

  • Создаёт отчёт в виде удобного файла с информацией о каждом тесте.

  • Можно встраивать скриншоты или другие вложения (например, при тестировании UI).

  • Подходит для использования в автоматических отчётах на CI.

4. pytest-mock
Упрощает работу с библиотекой unittest.mock, предоставляя встроенную фикстуру mocker.
Особенности:

  • Позволяет легко подменять объекты или функции.

  • Упрощает проверку, вызывался ли тот или иной метод.

  • Поддерживает патчинг классов, методов, функций и объектов.
    Фикстура mocker автоматически доступна в тестах после установки плагина.

5. pytest-bdd

Плагин для тестирования в формате BDD (Behavior Driven Development). Позволяет писать сценарии на понятном человеку языке (Gherkin) и связывать их с Python-кодом.
Особенности:

  • Поддержка Gherkin-синтаксиса: Given, When, Then.

  • Отдельные .feature файлы описывают поведение системы.

  • В коде реализуются шаги, соответствующие описанию сценариев.

  • Позволяет теснее связать тестирование с бизнес-логикой.

  • Полезен при командной разработке, когда над требованиями работают тестировщики, разработчики и аналитики.

6. pytest-flask
Плагин для интеграции с Flask-приложениями. (Flask — это легковесный веб-фреймворк для Python, который помогает создавать веб-приложения.)
Особенности:

  • Предоставляет фикстуру client для отправки HTTP-запросов к тестируемому приложению.

  • Позволяет запускать тесты внутри приложения и работать с его внутренними данными.

  • Подходит для функционального и интеграционного тестирования.

  • Полезен при тестировании API и веб-интерфейсов на Flask. 

7. pytest-repeat
Позволяет запускать один и тот же тест несколько раз подряд.
Особенности:

  • Полезен для поиска нестабильных (флакки) тестов.

  • Можно указать количество повторений через параметр --count.

8. pytest-randomly
Перемешивает порядок запуска тестов при каждом прогоне.
Особенности:

  • Помогает выявить зависимости между тестами, если они не изолированы друг от друга.

  • Можно зафиксировать порядок случайности с помощью seed.

9. pytest-timeout
Устанавливает ограничение по времени на выполнение каждого теста.
Особенности:

  • Предотвращает зависание тестов.

  • Позволяет выявить медленные или бесконечно работающие участки кода.

  • Поддерживает настройку времени через параметры запуска или фикстуры.

10. pytest-freezegun
Интеграция с библиотекой freezegun для фиксации текущей даты и времени во время теста.
Особенности:

  • Позволяет “заморозить” время, чтобы протестировать поведение, связанное с датами (например, истечение срока действия, расписания).

  • Удобно использовать с фикстурами.

  • Используется, если логика зависит от datetime.now() (возвращаение текущей даты) или аналогичных функций.

Для установки любого плагина используется команда pip install, например:
pip install pytest-cov pytest-xdist pytest-html pytest-mock pytest-bdd

Практическое задание

Цели задания:

  • Закрепить работу с ошибками и исключениями.
  • Понять применение фикстур с различным временем жизни.
  • Увидеть, как формируются отчёты.
  • Начать осваивать структуру тестов для реальных сценариев.

Есть файл auth.py со следующей логикой:

def get_user(username):
    users = {
        "admin": {"username": "admin", "password": "admin123"},
        "guest": {"username": "guest", "password": "guest"}
    }
    return users.get(username)

def login(username, password):
    user = get_user(username)
    if not user:
        raise ValueError("Пользователь не найден")
    if user["password"] != password:
        raise ValueError("Неверный пароль")
    return True

Нужно написать тесты в test_auth.py, реализуя следующее:

  1. Позитивный кейс: корректный логин пользователя.
  2. Негативные кейсы: пользователь не найден; пароль неверен.
  3. Фикстура с scope="module", которая будет возвращать валидные данные пользователя.
  4. Параметризация для разных комбинаций логина и пароля (успешные и ошибочные).
